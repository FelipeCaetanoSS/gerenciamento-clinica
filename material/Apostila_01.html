2. Arquitetura Cliente-Servidor: O Modelo Fundamental da Web

2.1 O que é a arquitetura cliente-servidor?

Toda aplicação web moderna funciona sobre um modelo chamado arquitetura cliente-servidor. Nesse modelo, dois atores se comunicam pela internet:

  • Cliente: é quem faz as requisições. Normalmente é o navegador do usuário (Chrome, Firefox) ou um aplicativo móvel (Android, iOS). O cliente exibe as informações para o usuário e envia dados para o servidor.
  • Servidor: é quem processa as requisições. Recebe os dados do cliente, executa a lógica de negócio, consulta o banco de dados e devolve uma resposta.
Analogia do restaurante: O cliente é o garçom que anota seu pedido e traz o prato. O servidor é a cozinha — o cliente nunca vai até lá, só faz pedidos. A cozinha (backend) tem toda a lógica: como preparar cada prato, onde guardar os ingredientes, quem pode cozinhar o quê. O menu que você vê é o frontend; a cozinha que você não vê é o backend.

2.2 O papel do Frontend

O frontend é tudo que roda no navegador do usuário — a interface visual. Suas responsabilidades são:

  • Exibir informações de forma visual (HTML, CSS)
  • Capturar interações do usuário (cliques, formulários, gestos)
  • Fazer requisições ao servidor via HTTP e exibir as respostas recebidas
  • Nunca conter lógica de negócio sensível ou validação definitiva (um usuário mal-intencionado pode manipular o frontend)

Tecnologias típicas do frontend: HTML, CSS, JavaScript, React, Vue, Angular.

2.3 O papel do Backend

O backend é o coração da aplicação. Suas responsabilidades são:

  • Receber e processar requisições do cliente (via HTTP)
  • Validar todos os dados recebidos — nunca confiar no frontend
  • Aplicar as regras de negócio (ex: "só pode fazer pedido quem tem saldo suficiente")
  • Comunicar com o banco de dados — ler, criar, atualizar e deletar informações
  • Autenticar e autorizar usuários — "quem é você e o que pode fazer?"
  • Enviar respostas ao cliente no formato correto (geralmente JSON)
⚠️ Erro conceitual comum: Muitos iniciantes acham que validar dados no frontend é suficiente. NÃO É. O frontend pode ser manipulado por qualquer usuário com conhecimento mínimo de DevTools. O backend DEVE revalidar TUDO que recebe, sempre.

2.4 Como cliente e servidor se comunicam? O Protocolo HTTP

A comunicação entre cliente e servidor ocorre pelo protocolo HTTP (HyperText Transfer Protocol). Cada comunicação é composta por:

  • Requisição (Request): enviada pelo cliente ao servidor. Contém: método (GET, POST, PUT, DELETE), URL, cabeçalhos (headers) e, às vezes, um corpo (body) com dados.
  • Resposta (Response): enviada pelo servidor ao cliente. Contém: código de status (200, 404, 500...), cabeçalhos e, normalmente, um corpo com dados em JSON.

4. Nivelamento JavaScript ES6+: O que você precisa saber

Esta seção é o conteúdo mais denso do Encontro 01 e serve como nivelamento — todos os conceitos aqui são fundamentais para programar em Node.js. Mesmo que você já conheça alguns deles, leia com atenção: os exemplos são voltados para o contexto de backend.

4.1 Arrow Functions — Funções de Seta

O que são?

Arrow functions são uma sintaxe mais curta e moderna para declarar funções em JavaScript, introduzida no ES6 (2015). Além da sintaxe compacta, têm um comportamento diferente em relação ao this, o que as torna essenciais em programação assíncrona e callbacks.

Comparação: função tradicional vs arrow function

// ✦ FORMA TRADICIONAL (ES5) — ainda válida, mas verbosa
function somar(a, b) {
  return a + b;
}

// ✦ ARROW FUNCTION (ES6+) — sintaxe compacta
const somar = (a, b) => {
  return a + b;
};

// ✦ ARROW FUNCTION com retorno implícito (quando há só 1 expressão)
const somar = (a, b) => a + b;
// A chave {} e o return são omitidos — JavaScript entende que deve retornar a + b

// ✦ Com apenas 1 parâmetro: os parênteses são opcionais
const dobrar = n => n * 2;

// ✦ Sem parâmetros: parênteses são obrigatórios
const saudar = () => "Olá, mundo!";

console.log(somar(3, 4));  // → 7
console.log(dobrar(5));    // → 10
console.log(saudar());     // → "Olá, mundo!"

Onde são mais usadas no backend?

Arrow functions são especialmente úteis em callbacks — funções passadas como argumento para outras funções. Você vai usar isso o tempo todo em Node.js:

// ✦ Filtrando um array de usuários ativos (backend frequentemente processa listas assim)
const usuarios = [
  { nome: "Ana", ativo: true },
  { nome: "Bob", ativo: false },
  { nome: "Carlos", ativo: true },
];

// Com função tradicional:
const ativos1 = usuarios.filter(function(u) { return u.ativo; });

// Com arrow function (mais limpo e idiomático):
const ativos2 = usuarios.filter(u => u.ativo);

// ✦ Extraindo apenas os nomes:
const nomes = usuarios.map(u => u.nome);
// → ["Ana", "Bob", "Carlos"]
⚠️ Atenção — Arrow function e this: Arrow functions não possuem seu próprio this. Elas herdam o this do contexto externo onde foram criadas. Isso as torna ideais para callbacks, mas inapropriadas para métodos de objetos que precisam referenciar o próprio objeto via this. Em Node.js puro, você raramente vai se preocupar com isso nos exemplos desta UC — mas é bom saber.
Boas práticas profissionais: Use arrow functions para callbacks, .map(), .filter(), .forEach() e funções anônimas curtas. Use function tradicional quando precisar de hoisting (função disponível antes da declaração) ou quando o this for relevante para o contexto do objeto.

4.2 Destructuring — Desestruturação

O que é?

Destructuring é uma sintaxe que permite extrair valores de arrays ou propriedades de objetos de forma direta, sem precisar acessar um por um. No backend, isso é usado constantemente para extrair dados de requisições HTTP, respostas de banco de dados e configurações.

Destructuring de Objetos

// ✦ Sem destructuring (modo antigo — verboso):
const usuario = { nome: "Ana", email: "ana@email.com", idade: 28 };
const nome  = usuario.nome;
const email = usuario.email;

// ✦ Com destructuring (modo moderno — limpo e direto):
const { nome, email, idade } = usuario;
console.log(nome);  // → "Ana"
console.log(email); // → "ana@email.com"

// ✦ Renomeando variáveis durante o destructuring:
const { nome: nomeDoUsuario, email: enderecoEmail } = usuario;
console.log(nomeDoUsuario); // → "Ana"

// ✦ Valores padrão (default) — caso a propriedade não exista:
const { nome, role = "user" } = usuario;
// Se 'role' não existir no objeto, assume "user" como padrão
console.log(role); // → "user"

// ✦ Exemplo real de backend — extraindo dados do body de uma requisição HTTP:
// req.body seria algo como: { nome: "Ana", email: "ana@email.com", senha: "123" }
const { nome, email, senha } = req.body;
// Agora temos as 3 variáveis prontas para usar!

Destructuring de Arrays

// ✦ Arrays usam [] em vez de {}
const frutas = ["maçã", "banana", "laranja"];

// Extrai na ordem: primeira variável = primeiro elemento
const [primeira, segunda] = frutas;
console.log(primeira); // → "maçã"
console.log(segunda);  // → "banana"

// ✦ Pulando elementos com vírgula:
const [, , terceira] = frutas;
console.log(terceira); // → "laranja"

// ✦ Exemplo real — troca de variáveis sem variável auxiliar:
let a = 1, b = 2;
[a, b] = [b, a];  // Troca elegante!
console.log(a, b); // → 2 1

Destructuring em parâmetros de função

// ✦ Muito comum em Express.js — extrair diretamente no parâmetro:

// Sem destructuring (menos legível):
function criarUsuario(dados) {
  console.log(dados.nome, dados.email);
}

// Com destructuring no parâmetro (mais legível e profissional):
function criarUsuario({ nome, email, senha }) {
  console.log(nome, email);
  // A senha ficou disponível mas não foi usada aqui — boas práticas!
}

// ✦ Em rotas Express (você vai ver isso muito):
// app.post('/usuarios', (req, res) => {
//   const { nome, email, senha } = req.body;  ← destructuring do body
//   const { id } = req.params;                ← destructuring dos params
// })
⚠️ Erro comum: Tentar fazer destructuring de null ou undefined causa um erro em tempo de execução: TypeError: Cannot destructure property 'nome' of undefined. Sempre valide se o objeto existe antes de desestruturar em código de produção.

4.3 Spread e Rest Operators — O Operador ...

O que são?

O operador ... (três pontos) tem dois usos distintos dependendo do contexto:

  • Spread (expansão): expande um array ou objeto em elementos individuais
  • Rest (agrupamento): agrupa múltiplos elementos em um único array ou objeto

Spread — expandindo dados

// ✦ Spread em Arrays — expandir os elementos de um array
const nums1 = [1, 2, 3];
const nums2 = [4, 5, 6];

// Sem spread (concat antigo):
const todos1 = nums1.concat(nums2); // → [1, 2, 3, 4, 5, 6]

// Com spread (mais elegante):
const todos2 = [...nums1, ...nums2]; // → [1, 2, 3, 4, 5, 6]

// ✦ Spread em Objetos — copiar/mesclar objetos (MUITO usado em backend)
const usuarioBase = { nome: "Ana", email: "ana@mail.com" };
const dadosExtras  = { role: "admin", ativo: true };

// Mesclar dois objetos em um novo objeto:
const usuarioCompleto = { ...usuarioBase, ...dadosExtras };
// → { nome: "Ana", email: "ana@mail.com", role: "admin", ativo: true }

// ✦ Criar cópia de um objeto com uma propriedade alterada (imutabilidade):
const usuarioAtualizado = { ...usuarioBase, nome: "Ana Paula" };
// → { nome: "Ana Paula", email: "ana@mail.com" }
// O objeto original 'usuarioBase' NÃO foi alterado!

// ✦ Exemplo real de backend — ao atualizar um usuário no banco:
// const usuarioAtualizado = await prisma.usuario.update({
//   where: { id },
//   data: { ...dadosAtuais, ...novosDados }  ← spread para mesclar
// })

Rest — agrupando parâmetros

// ✦ Rest em funções — captura todos os argumentos extras em um array
function somarTodos(...numeros) {
  // 'numeros' é um array com todos os argumentos recebidos
  return numeros.reduce((total, n) => total + n, 0);
}
somarTodos(1, 2, 3, 4, 5); // → 15

// ✦ Rest em destructuring — pegar o "resto" das propriedades
const usuario = { nome: "Ana", email: "ana@mail.com", senha: "hash123", role: "admin" };

// Separar a senha do restante (NUNCA retornar a senha em responses de API!):
const { senha, ...dadosPublicos } = usuario;
console.log(dadosPublicos); // → { nome: "Ana", email: "ana@mail.com", role: "admin" }
console.log(senha);         // → "hash123" (guardado separado, não será enviado ao cliente)
Padrão profissional muito importante: O exemplo do rest acima (const { senha, ...dadosPublicos } = usuario) é um padrão que você vai usar constantemente ao retornar dados de usuários. Nunca retorne a senha (mesmo hasheada) em respostas de API. Separe ela com rest antes de enviar a resposta.

4.4 Promises — Promessas Assíncronas

O problema: operações que levam tempo

Em qualquer aplicação backend, operações como leitura de banco de dados, chamadas a APIs externas ou leitura de arquivos são assíncronas — elas levam tempo e não bloqueiam a execução do programa. JavaScript precisa de um mecanismo para lidar com isso: as Promises.

Analogia: Você vai a uma lanchonete, faz o pedido e recebe uma senha numérica. Você não fica parado na fila esperando — vai sentar, conversa, faz outras coisas. Quando o pedido fica pronto, te chamam. A senha numérica é a Promise: uma "promessa" de que algo será entregue no futuro, podendo terminar em sucesso (.then) ou fracasso (.catch).

O que é uma Promise?

Uma Promise é um objeto que representa a eventual conclusão (ou falha) de uma operação assíncrona. Ela sempre está em um de três estados:

  • Pending (pendente): a operação ainda está em andamento
  • Fulfilled (cumprida): a operação concluiu com sucesso
  • Rejected (rejeitada): a operação falhou

Criando e consumindo Promises

// ✦ Criando uma Promise manualmente (você fará isso raramente — geralmente usa prontas)
const minhaPromise = new Promise((resolve, reject) => {
  // 'resolve' é chamado quando a operação tem SUCESSO
  // 'reject' é chamado quando a operação FALHA

  const sucesso = true; // Simulando resultado

  if (sucesso) {
    resolve("Operação bem sucedida!"); // Promise fulfilled
  } else {
    reject(new Error("Algo deu errado")); // Promise rejected
  }
});

// ✦ Consumindo uma Promise com .then() e .catch()
minhaPromise
  .then(resultado => {
    // Executado quando a Promise é fulfilled
    console.log(resultado); // → "Operação bem sucedida!"
  })
  .catch(erro => {
    // Executado quando a Promise é rejected
    console.error("Erro:", erro.message);
  })
  .finally(() => {
    // Executado sempre, com sucesso ou falha (ex: fechar loading spinner)
    console.log("Operação finalizada");
  });

Promise.all — executar várias Promises em paralelo

// ✦ Imagine que você precisa buscar usuário E seus pedidos ao mesmo tempo
// Em vez de esperar um e depois o outro (sequencial e lento),
// execute ambos em paralelo com Promise.all:

const buscarUsuario = () => Promise.resolve({ id: 1, nome: "Ana" });
const buscarPedidos = () => Promise.resolve([{ id: 10, produto: "Notebook" }]);

Promise.all([buscarUsuario(), buscarPedidos()])
  .then(([usuario, pedidos]) => {
    // Ambos terminaram — os resultados chegam como array na mesma ordem
    console.log(usuario);  // → { id: 1, nome: "Ana" }
    console.log(pedidos);  // → [{ id: 10, produto: "Notebook" }]
  });

// ⚠️ Se QUALQUER uma das Promises falhar, Promise.all vai para o .catch()
⚠️ O "callback hell" — por que Promises existem: Antes das Promises, todo código assíncrono usava callbacks aninhados. Com múltiplas operações em sequência, o código ficava profundamente indentado (o famoso "pyramid of doom"). Promises resolveram isso com o encadeamento de .then(), e o async/await (próxima seção) tornou o código ainda mais legível.

4.5 async/await — A forma moderna de escrever código assíncrono

O que são?

async/await é uma sintaxe construída sobre Promises que permite escrever código assíncrono como se fosse síncrono — muito mais legível e fácil de debugar. É o padrão dominante no desenvolvimento Node.js moderno.

  • async: declara que uma função é assíncrona. Toda função async retorna automaticamente uma Promise.
  • await: usado dentro de funções async. Pausa a execução da função até que a Promise seja resolvida, sem bloquear o restante do programa.
// ✦ Comparação: .then() vs async/await para a mesma operação

// === COM .then() (ainda válido, mas menos legível em sequências) ===
function buscarDadosThen() {
  buscarUsuario(id)
    .then(usuario => buscarPedidosDoUsuario(usuario.id))
    .then(pedidos => console.log(pedidos))
    .catch(erro => console.error(erro));
}

// === COM async/await (muito mais legível — padrão de mercado) ===
async function buscarDados(id) {
  const usuario = await buscarUsuario(id);   // Aguarda o usuário
  const pedidos  = await buscarPedidos(usuario.id); // Depois aguarda os pedidos
  console.log(pedidos);
}

// ✦ Tratamento de erros com async/await: use try/catch
async function buscarDadosSeguro(id) {
  try {
    const usuario = await buscarUsuario(id);
    const pedidos  = await buscarPedidos(usuario.id);
    console.log("Pedidos encontrados:", pedidos);
  } catch (erro) {
    // Qualquer erro no bloco try cai aqui
    console.error("Erro ao buscar dados:", erro.message);
  }
}

await com Promise.all — o melhor dos dois mundos

// ✦ Combine async/await com Promise.all para operações paralelas:
async function carregarPaginaInicial(usuarioId) {
  // Executa as 3 buscas em PARALELO (não sequencial) — muito mais rápido!
  const [usuario, pedidos, notificacoes] = await Promise.all([
    buscarUsuario(usuarioId),
    buscarPedidos(usuarioId),
    buscarNotificacoes(usuarioId),
  ]);

  return { usuario, pedidos, notificacoes };
}
⚠️ Erro muito comum — await em loop: Usar await dentro de um for...of é sequencial (um aguarda o outro). Se quiser paralelismo, use Promise.all(array.map(...)). Executar 10 queries sequencialmente quando poderiam ser paralelas é um problema de performance grave em produção.

4.6 Módulos ES — import e export

O que são módulos?

Em projetos reais, você não coloca todo o código em um único arquivo — isso seria impossível de manter. O sistema de módulos permite dividir o código em arquivos independentes, cada um com uma responsabilidade clara, e depois conectá-los com import e export.

Por que isso importa no mercado? Em uma empresa, vários desenvolvedores trabalham no mesmo projeto. Sem módulos, editar um arquivo significaria conflito constante com os colegas. Com módulos bem definidos, cada desenvolvedor edita sua "peça" sem interferir nas demais.

CommonJS (CJS) vs ES Modules (ESM)

Existem dois sistemas de módulos no mundo Node.js:

CaracterísticaCommonJS (CJS)ES Modules (ESM)
Sintaxe para importarrequire()import
Sintaxe para exportarmodule.exports =export / export default
Quando surgiuNode.js original (2009)ES6 (2015), nativo no browser
Uso hojeMuitos projetos legadosPadrão moderno recomendado
Extensão do arquivo.js (com CJS config).mjs ou .js (com "type":"module")

CommonJS — require (você vai encontrar isso em código legado)

// === ARQUIVO: matematica.js (CommonJS) ===
function somar(a, b) { return a + b; }
function subtrair(a, b) { return a - b; }

module.exports = { somar, subtrair }; // Exporta um objeto

// === ARQUIVO: app.js ===
const { somar, subtrair } = require("./matematica"); // Importa do arquivo
console.log(somar(2, 3)); // → 5

ES Modules — import/export (padrão moderno que usaremos)

// ✦ EXPORTS NOMEADOS — exporte múltiplas funções/variáveis do mesmo arquivo

// === ARQUIVO: utils/matematica.js ===
export function somar(a, b) { return a + b; }
export function subtrair(a, b) { return a - b; }
export const PI = 3.14159;

// === ARQUIVO: app.js ===
import { somar, PI } from "./utils/matematica.js";
// Importa apenas o que precisa — mais eficiente!
console.log(somar(2, 3)); // → 5
console.log(PI);           // → 3.14159

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

// ✦ DEFAULT EXPORT — quando o arquivo exporta uma única coisa principal

// === ARQUIVO: services/usuarioService.js ===
class UsuarioService {
  async criar(dados)   { /* lógica */ }
  async buscarPorId(id) { /* lógica */ }
}

export default new UsuarioService(); // Exporta instância da classe

// === ARQUIVO: controllers/usuarioController.js ===
import usuarioService from "../services/usuarioService.js";
// No default export: qualquer nome pode ser usado no import!
import meuServico from "../services/usuarioService.js"; // Também funciona

// ✦ Importar módulos externos (pacotes NPM):
import express from "express";          // Sem './' = pacote instalado (NPM)
import { Router } from "express";       // Named import de pacote NPM
import path from "node:path";          // Módulo nativo do Node (prefixo node:)

Estrutura de projeto usando módulos — como funciona na prática

// Estrutura típica de um projeto Express com módulos ES:
//
// src/
// ├── app.js              ← configura o Express, middlewares globais
// ├── server.js           ← inicia o servidor (listen)
// ├── routes/
// │   ├── index.js        ← agrupa todas as rotas
// │   └── usuarios.js     ← rotas de /usuarios
// ├── controllers/
// │   └── usuarioController.js  ← lida com req/res
// ├── services/
// │   └── usuarioService.js     ← regras de negócio
// └── utils/
//     └── matematica.js         ← funções auxiliares

// Cada arquivo exporta exatamente o que é necessário,
// e importa apenas o que precisa dos outros. Limpo e escalável.
⚠️ Não esqueça a extensão .js! Em ES Modules, você deve incluir a extensão no caminho do import: import somar from './matematica.js'. Sem o .js, o Node lança um erro. Isso é diferente do que acontece no webpack/bundlers que resolvem automaticamente.
Habilitando ES Modules no Node.js: Adicione "type": "module" no seu package.json. Com isso, todos os arquivos .js do projeto são tratados como ESM. Se precisar de um arquivo CJS específico, renomeie para .cjs.

// package.json
{ "type": "module" }
